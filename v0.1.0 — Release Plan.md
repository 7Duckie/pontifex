# Pontifex v0.1.0 — Release Plan

- **Status:** Local working document (not committed to repo)
- **Scope locked:** 2026-05-21
- **Last reviewed:** 2026-05-21

## Purpose

This document defines what v0.1.0 of Pontifex ships, why, in what order,
and against what acceptance criteria. It is the single reference the
project consults during the v0.1.0 build: when sequencing a commit,
when checking whether a PR is in scope, when deciding whether the
release is ready to tag.

It is a *plan*, not a *spec*. The format spec (`ARCHIVE-FORMAT.md`)
describes what the code must produce on disk. This document describes
what code we are going to write to produce it, in what order, and what
we are deliberately not writing yet. When the two appear to disagree,
the format spec governs the bytes and this document governs the
schedule.

The document lives locally rather than in the repo. v0.1.0 is the first
basic release; the ethos is to keep ceremony low and surface area
small. Working plans churn during the build, and most of those churns
do not deserve to be in the project's permanent record. What does
deserve to be in the record — the release notes, the CHANGELOG entry,
any spec amendments the build surfaces, deferral entries against
`IDEA-BANK.md` — gets committed on its own schedule.

## Scope

### In scope for v0.1.0

The headline feature is **a complete round trip**: a user can run
`wp pontifex export <path>` on a real WordPress site, take the
resulting `.wpmig` file to a different machine, run
`wp pontifex import <path>` against a fresh WordPress installation,
and end up with a site that matches the source.

Concretely:

- `wp pontifex export <path>` — WP-CLI command that produces a
  `.wpmig` archive from the current site.
- `wp pontifex import <path>` — WP-CLI command that restores a
  `.wpmig` archive onto the current site.
- `ArchiveWriter` and `ArchiveReader` libraries conforming to the v1
  byte layout described in `ARCHIVE-FORMAT.md`.
- Codec `0x0000` (no compression, no encryption) as the mandatory
  baseline.
- Codec `0x0001` (gzip, no encryption). PHP ships gzip and the cost
  of including it is negligible.
- All four entry kinds the format defines: `file`, `db_chunk`,
  `directory`, `symlink`.
- The full integrity contract for unencrypted archives: per-entry
  SHA-256, manifest hash, footer manifest hash, mandatory verification
  order on read.
- Exceptions with clear messages for everything that can go wrong,
  surfaced by WP-CLI's normal error handling and `-v`/`-vv`/`-vvv`
  verbosity controls.
- A transient summary line at the end of each command
  ("Exported 1,234 files, 4,567,890 bytes in 12.3 seconds"). No
  persistence, no logging file, no `wp_options` writes for stats.
- Round-trip tests proving write-then-read reconstructs the source
  byte-perfectly.
- WordPress integration tests across PHP 8.1–8.4 and at least two
  WordPress versions, running in CI.
- Documentation: README updates, command help text, one worked
  example per command, an explicit statement that v0.1.0 archives are
  not encrypted.

### Explicitly deferred from v0.1.0

Deferred items live here with their reasoning, so they do not drift
back into scope without a decision.

- **Encryption** (codecs `0x0100` / `0x0101` / `0x0102`; Argon2id KDF;
  AES-256-GCM cipher) → **v0.2.0**. The format spec's "encrypt by
  default" promise is a property of v1.0 of the *format*; the *plugin*
  delivers it in stages. v0.1.0 archives set flag bit 0 to zero, and
  the footer's 16-byte `argon2id_salt` field is zero-filled. This is
  the largest scope cut and the release notes must call it out:
  v0.1.0 archives deposited into untrusted storage are not protected
  by encryption.

- **Zstd codec** (`0x0002`) → **v0.2.0**. PHP's zstd extension is not
  universally available; adding it means an environment check we would
  rather defer. Gzip handles compression for v0.1.0.

- **Ed25519 signatures** (flag bit 1; optional signature block after
  the footer) → **v0.2.0**. The cryptographic dependency profile
  lands together with encryption.

- **Transfer-metric counters** (IDEA-BANK Idea 002) → **v0.2.0**.
  Reconsidered at scope lock: v0.1.0 does not need persistent metrics
  to prove the round trip works. The transient summary printed at the
  end of each command is sufficient. The IDEA-BANK entry should be
  updated to record this deferral.

- **PSR-3 logger and per-transfer log files** (IDEA-BANK Idea 003) →
  **v0.2.0**. Same reasoning. v0.1.0 surfaces failures through
  exceptions and WP-CLI verbosity, not a structured logging
  subsystem.

- **`wp pontifex stats` CLI command** → **v0.2.0–v0.3.0**, per
  IDEA-BANK Idea 002.

- **`wp pontifex diagnostics` CLI command** → **v0.2.0–v0.3.0**, per
  IDEA-BANK Idea 003.

- **The Go CLI verifier** → **v2.0**.

- **Admin UI of any kind** → **v0.3.0+**. v0.1.0 is WP-CLI only.

- **Plugin install telemetry** → **deferred indefinitely**, per
  IDEA-BANK Idea 001.

## System design

The codebase grows from one production-bearing class (`DoctorCommand`)
to roughly 15–20 classes. The discipline established in v0.0.x is
unchanged: every class testable in isolation through interfaces, every
WordPress dependency routed through the `Environment` seam, no
statics, no globals, no `setAccessible` tricks.

### Module layout

The proposed directory structure under `src/`:

```
src/
├── Environment/              (exists)
│   ├── Environment.php       (interface)
│   └── WpEnvironment.php     (real implementation)
├── Command/                  (exists; DoctorCommand lives here)
│   ├── DoctorCommand.php
│   ├── ExportCommand.php     (new)
│   └── ImportCommand.php     (new)
├── Archive/
│   ├── Format/
│   │   ├── Header.php              (value object: magic, version, flags)
│   │   ├── Provenance.php          (value object: WP version, PHP version, etc.)
│   │   ├── EntryHeader.php         (value object: kind, path, codec, lengths)
│   │   ├── ArchiveManifest.php     (value object: list of EntryHeader with offsets)
│   │   ├── Footer.php              (value object: manifest offset/length/hash/salt)
│   │   └── ByteOrder.php           (big-endian read/write helpers)
│   ├── Writer/
│   │   ├── ArchiveWriter.php       (orchestrator)
│   │   ├── EntryWriter.php
│   │   ├── ManifestWriter.php
│   │   └── FooterWriter.php
│   ├── Reader/
│   │   ├── ArchiveReader.php       (orchestrator)
│   │   ├── EntryReader.php
│   │   ├── ManifestReader.php
│   │   └── FooterReader.php
│   ├── Codec/
│   │   ├── Codec.php               (interface: encode/decode streams)
│   │   ├── RawCodec.php            (codec 0x0000)
│   │   ├── GzipCodec.php           (codec 0x0001)
│   │   └── CodecRegistry.php       (maps codec ID → Codec)
│   └── Integrity/
│       ├── HashingStream.php       (wraps a stream; computes SHA-256 in flight)
│       └── Sha256.php              (thin wrapper; single point of dependence)
├── Manifest/                       (NB: distinct from Archive\Format\ArchiveManifest)
│   ├── SourceManifest.php          (value object: what the writer is about to write)
│   ├── ManifestBuilder.php         (walks WordPress to build a SourceManifest)
│   ├── FileScanner.php             (enumerates wp-content files)
│   ├── DatabaseScanner.php         (chunks wpdb tables into db_chunks)
│   └── ExclusionRules.php          (cache dirs, .git, node_modules, etc.)
└── Importer/
    ├── Importer.php                (orchestrator)
    ├── FileApplier.php             (restores file entries)
    ├── DatabaseApplier.php         (replays db_chunk entries via wpdb)
    ├── UrlRewriter.php             (handles serialized PHP for URL rewriting)
    └── CompatibilityCheck.php      (compares provenance against current site)
```

### Key seams

The interfaces that make testing possible without booting WordPress:

1. **`Environment`** — already exists. Will gain methods like
   `list_files(string $root): iterable`,
   `wpdb_tables(): array`, `get_option(string $name)`,
   `query(string $sql): iterable`. Every WordPress dependency the
   archive code needs is routed through here, never invoked directly.

2. **`Codec`** — `encode(resource $in, resource $out): int` and the
   matching `decode`. Stream-in, stream-out by design — the
   alternative (string-in, string-out) defeats the streaming
   architecture and breaks down on hosts with PHP memory caps.

3. **Stream handles.** PHP's `php://memory` for unit tests, real file
   handles in production. No abstraction needed beyond PHP's native
   stream API — adding one would be over-engineering at v0.1.0 scale.

### Namespace clash to resolve before Phase 1

`Archive\Format\ArchiveManifest` (the bytes of the manifest block) and
`Manifest\SourceManifest` (the plan for what the writer is about to
encode) are different concerns. The provisional naming above is fine
but worth one pass before commit 1 to make sure each name reads
naturally where it appears. The convention "Archive… for byte-layout
concerns, plain names for in-memory concerns" works as a default.

## Data flow

### Export

When the user runs `wp pontifex export ./site.wpmig`:

1. `ExportCommand` validates the output path is writable.
2. `ExportCommand` fires action `pontifex_before_export` (extension
   point for other plugins).
3. `Environment` gathers provenance: WP version, PHP version, site
   URL, charset, DB collation, plugin version, generator string,
   ISO-8601 timestamp with explicit UTC offset.
4. `Manifest\ManifestBuilder` walks the site:
  - `FileScanner` enumerates `wp-content/`, honouring `ExclusionRules`
    (cache directories, version-control metadata, `node_modules`,
    and so on).
  - `DatabaseScanner` enumerates `wpdb` tables and chunks rows into
    `db_chunk` entries. Chunk boundaries respect a memory budget so
    large tables stream rather than materialise.
  - Returns a `SourceManifest` — the plan for what to write.
5. `ArchiveWriter` opens the output stream and writes:
  - 16-byte header: magic `WPMIG\0\0\x01`, format version, flags=0.
  - Provenance block: 4-byte length, 32-byte SHA-256, JSON payload.
  - For each entry in the `SourceManifest`:
    - `EntryWriter` writes the entry header (length-prefixed JSON).
    - `EntryWriter` writes codec ID (2 bytes) and nonce
      (12 bytes, zero-filled in v0.1.0 since no encryption).
    - The `Codec` streams the payload from source → output, with a
      `HashingStream` tee computing SHA-256 as bytes pass through.
    - `EntryWriter` writes the 32-byte hash at the end of the entry.
    - The entry's offset and length are recorded in an in-memory
      `ArchiveManifest`.
  - `ManifestWriter` writes the `ArchiveManifest` block
    (length + hash + JSON).
  - `FooterWriter` writes the 64-byte footer: manifest offset,
    manifest length, manifest hash, 16-byte zero salt.
6. The stream is closed and flushed to disk.
7. `ExportCommand` fires action `pontifex_after_export`.
8. The command prints its transient summary and exits 0.

### Import

When the user runs `wp pontifex import ./site.wpmig`:

1. `ImportCommand` validates the archive exists and is readable.
2. `ArchiveReader` opens the input stream and:
  - Reads the 16-byte header; validates magic and version.
  - Reads the provenance block; verifies SHA-256; parses JSON.
  - Seeks to end of file; reads the 64-byte footer.
  - Reads the `ArchiveManifest` using the footer's offset and
    length; verifies its hash matches the footer.
  - Returns a reader handle ready to stream entries.

   The verification order is non-negotiable. The spec calls it out as
   a security invariant; reordering creates exploitable races. The
   reader's public API must make it impossible to read entry payloads
   before the footer and manifest have been validated.

3. `CompatibilityCheck` compares the archive's provenance against the
   current `Environment`:
  - Refuses on incompatible WordPress major version (with `--force`
    override that always leaves a persistent admin notice).
  - Warns on PHP version mismatch.
  - Warns on charset / collation mismatch.
4. If `--dry-run` is set, the command prints the planned actions and
   exits.
5. `ImportCommand` fires action `pontifex_before_import`.
6. For each entry in the manifest:
  - `EntryReader` streams the payload through the `Codec`, with a
    `HashingStream` tee verifying the SHA-256 against the stored hash.
  - The decoded payload is dispatched by entry kind:
    - `file` → `FileApplier` writes to `wp-content/`.
    - `db_chunk` → `DatabaseApplier` replays INSERT / REPLACE
      statements through `wpdb`.
    - `directory` → `FileApplier` creates the directory.
    - `symlink` → `FileApplier` creates the symlink, refusing any
      link that resolves outside the site root.
7. If `--rewrite-urls=old,new` is supplied, `UrlRewriter` walks
   `wp_options`, `post_meta`, and `post_content`, replacing URLs while
   preserving serialized PHP integrity.
8. `ImportCommand` fires action `pontifex_after_import`.
9. The command prints its transient summary and exits 0.

## Test architecture

Three layers, each with a distinct job. Each layer's tests must be
runnable independently and pass in CI.

**Layer 1: unit tests (brain/monkey).** Each class tested in isolation
with mocked dependencies. Fastest, runs on every save. Covers
byte-layout correctness, codec encode/decode, hash computation,
manifest serialisation, value-object invariants, command argument
parsing. The bulk of tests live here — probably 80–150 by end of
v0.1.0.

**Layer 2: round-trip tests (unit-level, no WordPress).** The cheapest
and strongest correctness check we have. Build a synthetic
`SourceManifest` in memory (a few files, a fake `db_chunk`, a
`directory` entry). Write via `ArchiveWriter` to a `php://memory`
stream. Read back via `ArchiveReader`. Assert byte-identical
reconstruction of every entry. Run against every codec, every entry
kind, every edge case (empty files, large streamed files, multi-byte
filenames, NULL values in rows). Probably 30–50 tests. No WordPress,
no filesystem, runs in milliseconds.

**Layer 3: WordPress integration tests (wp-phpunit).** Real WordPress,
real database. Boot a fixture site, populate it with content, run
`wp pontifex export` via WP-CLI, run `wp pontifex import` against a
fresh site, assert the imported site matches the source. Slow but the
ground truth. Probably 5–15 tests by end of v0.1.0 — enough to cover
the happy path and obvious failure modes, not exhaustive.

## Integration test infrastructure

Layer 3 needs setup that v0.0.x deferred. What is required:

- A WordPress installation tests can boot against. Download into
  `tests/wp/` at setup time. Standard approach, slow first run, fast
  after. Docker is a possible alternative but not necessary at this
  scale.
- A database. SQLite via the SQLite Database Integration plugin for
  speed locally; MySQL via GitHub Actions service container in CI.
- A test fixture site — a known set of posts, pages, options, and
  files. A small PHP script populates it at test boot. Fixture must
  include the corner cases v0.1.0 must handle: a serialised array in
  `wp_options`, a multi-byte filename, a file larger than the default
  PHP memory limit, a symlink where the OS supports it, a row with a
  NULL value.
- WP-CLI available in CI for end-to-end tests. GitHub Actions has a
  `setup-wp-cli` action.

The PHPUnit 10 + wp-phpunit compatibility issue noted during v0.0.x
remains. Budget half a day for setup, up to a day if it goes badly.

## "Done" criteria for v0.1.0

A v0.1.0 release is ready to tag when, and only when:

1. A user can run `wp pontifex export /tmp/site.wpmig` on a real
   WordPress site and the command exits 0 with a file at the named
   path.
2. The same user can run `wp pontifex import /tmp/site.wpmig` on a
   fresh WordPress installation and the resulting site matches the
   source for: all files in `wp-content/`, all rows in all `wpdb`
   tables, all options in `wp_options`.
3. The round-trip test suite (Layer 2) passes 100%.
4. The integration test suite (Layer 3) passes across the full CI
   matrix.
5. `composer check` is green: PHPCS, PHPStan level 6, PHPUnit, audit.
6. README documents the two commands with at least one worked example
   each.
7. The release notes call out explicitly that v0.1.0 archives are
   unencrypted and that encryption arrives in v0.2.0.
8. `ARCHIVE-FORMAT.md` is updated to v1.0 (locked) with any
   build-driven amendments folded in.
9. `IDEA-BANK.md` has deferral entries appended to Ideas 002 and 003.
10. The v0.1.0 tag is signed, pushed, and a GitHub release is
    published.

## Risks

In rough order of how much each could hurt:

**URL rewriting in serialised PHP.** WordPress stores serialised
arrays everywhere — `wp_options`, `post_meta`, theme and plugin
settings. Naïve string-replace on serialised data corrupts it (length
prefixes go wrong). A proper serialised-PHP walker is required. WP-CLI's
`search-replace` command has one we can study or extract. This is the
single commit most likely to overrun its estimate.

**wp-phpunit setup.** Past sessions flagged this. Half a day is the
optimistic estimate; a day if it goes badly. Mitigation: schedule it
early in Phase 6 so problems are discovered before Phase 7.

**Spec amendments.** The format spec is DRAFT for a reason. Writing
implementation against it will surface ambiguities. A spec amendment
commit is normal work, not a setback. Budget time for it.

**Database export at scale.** A WordPress site with millions of
`post_meta` rows produces a huge `db_chunk` if chunk boundaries are
wrong. `DatabaseScanner` must stream via `wpdb`'s `get_results` with
explicit `LIMIT` / `OFFSET` paging, never materialising whole tables
in memory.

**Symlink safety on import.** Restoring symlinks blindly is a
directory-traversal risk if an archive is crafted maliciously. v0.1.0
refuses symlinks that resolve outside the site root. Acceptable for
v0.1.0; v0.2.0 may add opt-in for trusted archives.

**PHP memory limits during import.** Hosts cap PHP at 128–512 MiB.
Nothing can load a whole entry into memory — everything streams. The
`Codec` interface is stream-in, stream-out, not string-in,
string-out. This is an invariant we enforce by design from the first
commit.

**Thin diagnostics in v0.1.0.** Without a structured logger or
per-transfer log file, bug reports against v0.1.0 will be harder to
triage than they will be from v0.2.0 onward. This is a known and
accepted trade-off. Bug reporters can use WP-CLI verbosity
(`-v` / `-vv` / `-vvv`) to capture more context.

## Open decisions

Items still to settle, ideally before Phase 1 commit 1:

1. **Namespace naming.** `Archive\Format\ArchiveManifest` vs
   `Manifest\SourceManifest` — confirm the convention reads naturally
   at every callsite.
2. **Maximum `db_chunk` size.** A working default in bytes
   (e.g. 4 MiB per chunk) needs picking. Affects memory profile of
   both writer and reader.
3. **`ExclusionRules` defaults.** Which directories to skip by
   default. Cache directories obviously; what else? `node_modules`,
   `.git`, common backup-plugin output folders. A small list to ratify
   before Phase 2 commit 12.
4. **`db_chunk` table-order policy.** Tables with foreign-key
   dependencies must be applied in the right order. The format's
   `db_chunk` carries enough information to handle this on read, but
   the writer's emission order matters for streaming readers. Decide
   before Phase 2 commit 11.

## Next concrete action

Phase 1 commit 1: the `Codec` interface plus `RawCodec` plus tests. It
is small, self-contained, depends on nothing outside what already
exists, and forces the first design decision (the stream-in,
stream-out signature of `Codec::encode` and `Codec::decode`). Get that
signature right and every subsequent codec slots in cleanly; get it
wrong and every later commit pays interest.
